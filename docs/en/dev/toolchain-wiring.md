# Stage 21 — ToolchainInstaller wiring

> Date: 2026-06-18
> Status: completed
> Related stages: after Stage 20 (hybrid migration), before Stage 22 (final integration)

---

Stage 11 introduced `ToolchainInstaller` with SHA-256 verification but left
it unwired — the caller was expected to invoke it manually before any
firmware operation. Stage 21 closes that gap by automatically running the
installer when `FirmwareOperationService` is created.

## What this stage does

Updates `FirmwareOperationService.onCreate()` to:

1. Load the toolchain manifest from `assets/toolchain/manifest.json` via
   `ToolManifestLoader`.
2. Construct `ToolchainInstaller` with an `assetProvider` lambda that opens
   `assets/bin/<name>` streams.
3. Run the installer against `<filesDir>/toolchain-bin/` on a background
   coroutine (using `SupervisorJob` so the install is not cancelled when a
   single operation finishes).
4. Serialize concurrent installs via a `Mutex` — multiple
   `startForegroundService` calls cannot trigger parallel installs.
5. Stash the resulting `toolsDir` in `FirmwareOperationService.lastInstalledToolsDir`
   so `OperationExecutor` can populate the `TOOLS_DIR` env var.
6. Handle all three `ToolchainInstallResult` variants:
   - `Success` — set `lastInstalledToolsDir`.
   - `Failed` — log the message, keep the service running (the operation may
     still work if the tools are already on disk).
   - `ChecksumMismatch` — log the mismatch; the bad file was already deleted
     by the installer.

## Why onCreate (not onStartCommand)

`onCreate` is called exactly once per service lifetime. `onStartCommand` can
fire multiple times (one per `startForegroundService(...)` call). Putting the
install in `onCreate` guarantees:

- The install runs exactly once per service process.
- Subsequent `startForegroundService` calls do not re-trigger the install
  (they go through `onStartCommand`, which just builds the notification and
  returns `START_NOT_STICKY`).

The install itself is **idempotent** anyway — `ToolchainInstaller` skips
already-installed files after SHA-256 verification (when a checksum is
declared). The manifest currently ships with `sha256=null` for all entries,
so verification is off and the install is a fast existence check.

## New behavior

```text
FirmwareOperationService.onCreate()
  -> ensureToolchainInstalled()
       -> serviceScope.launch (SupervisorJob + Dispatchers.IO)
            -> installMutex.withLock
                 -> runInstallerIfNeeded()
                      -> loadManifest() from assets/toolchain/manifest.json
                      -> ToolchainInstaller(manifest, assetProvider).install(toolsDir)
                      -> when Success: lastInstalledToolsDir = toolsDir
                      -> when Failed/ChecksumMismatch: log and continue
```

## CI gate

```bash
python3 tools/check-hybrid-migration.py
```

The check verifies:

- `FirmwareOperationService.onCreate` overrides and calls
  `ensureToolchainInstalled`.
- All three `ToolchainInstallResult` variants are handled.
- `lastInstalledToolsDir` companion field is exposed.
- `SupervisorJob` + `installMutex` are used to make the install safe under
  concurrent service starts.

## What is intentionally NOT done here

- `OperationExecutor` does NOT yet read `FirmwareOperationService.lastInstalledToolsDir`
  to populate the `TOOLS_DIR` env var. The executor takes the tools dir via
  `toolsDirProvider`, so the wiring is a one-liner in the calling activity —
  deferred to a future stage.
- The manifest now ships with real SHA-256 values for every shipped binary
  (generated by `tools/compute-tool-hashes.py`). Two entries (`lpunpack`,
  `simg2img`) keep `sha256=null` because the binaries are not shipped in
  `assets/bin/` — they are declared in the manifest for completeness.
- The install runs with `verifyChecksums=false` because the legacy
  `LegacyShellBridge.installToolkit` path does not require checksum
  verification (it just needs the binaries on disk). `FirmwareOperationService`
  also uses `verifyChecksums=false` for the same reason. Once the
  `ToolchainInstaller` is the single installation path, the flag can be
  flipped to `true`.
- The install does NOT block `onStartCommand` — the first operation may
  start before the install finishes. This is acceptable because the legacy
  `LegacyShellBridge.installToolkit` path (Stage 23) extracts the binaries
  for the legacy operations; the installer only populates the new
  `toolsDir` for `OperationExecutor` to use.
