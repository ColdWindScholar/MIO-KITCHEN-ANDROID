# Stage 11 — ToolchainInstaller

> Date: 2026-06-18
> Status: completed
> Related stages: after Stage 10 (runtime profiles), before Stage 12 (OperationExecutor)

---

The Stage 10 manifest at `assets/toolchain/manifest.json` declares which
native tools the app ships with, including SHA-256 checksums. Stage 11 closes
the loop by introducing a `ToolchainInstaller` that actually extracts those
binaries into a writable directory and verifies them against the manifest.

## What this stage does

Before this stage there was no typed way to:

- check that a tool the operation plan needs is actually present on disk;
- verify that a bundled binary has not been corrupted or tampered with;
- re-extract a missing or corrupted tool at runtime;
- mark `ToolchainProfile.toolsDir` and `verifiedChecksums` with real values.

Stage 11 introduces:

- `ToolchainInstallResult` — sealed type covering `Success`, `Failed`, and
  `ChecksumMismatch`.
- `ToolchainInstaller` — extracts binaries from an `assetProvider`, verifies
  SHA-256 (when the manifest declares one), sets the executable bit, and
  reports per-tool success/skip.

## New API

```text
common/toolchain/
  ToolchainInstaller.kt     # sealed ToolchainInstallResult + ToolchainInstaller class

common/src/test/java/.../toolchain/
  ToolchainInstallerTest.kt  # 9 JVM tests using real temp files
```

## Contract

```text
ToolchainInstaller
  -> install(toolsDir, verifyChecksums, overwrite): ToolchainInstallResult
  -> does NOT run shell
  -> does NOT depend on Android Context
  -> uses an assetProvider lambda (testable in JVM)
  -> uses a sha256Verifier lambda (overridable in tests)
  -> sets executable bit on installed files
  -> deletes corrupted files on ChecksumMismatch

ToolchainInstallResult
  -> Success(toolsDir, installedTools, skippedTools, verifiedChecksums)
  -> Failed(message, cause, partialDir)
  -> ChecksumMismatch(toolName, expected, actual)
```

## Algorithm

1. Create `toolsDir` if it does not exist.
2. For each `ToolDescriptor` in the manifest:
   - If `toolsDir/<name>` already exists and `overwrite=false`:
     - Verify SHA-256 if declared (return `ChecksumMismatch` on failure).
     - Mark as installed; continue.
   - Open a stream from `assetProvider(name)`.
   - If the stream is `null`, skip and record in `skippedTools`.
   - Copy the stream to `toolsDir/<name>`.
   - If `verifyChecksums=true` and `sha256` is declared, verify.
     - On mismatch, delete the file and return `ChecksumMismatch`.
   - Set the executable bit (`setExecutable(true, true)`).
3. Return `Success` with the installed/skipped lists.

## CI gate

```bash
python3 tools/check-toolchain-installer.py
```

Expected output:

```text
PASS: ToolchainInstaller with SHA-256 verification is in place
PASS: ToolchainInstallResult sealed type covers Success/Failed/ChecksumMismatch
PASS: ToolchainInstaller tests cover install/skip/overwrite/checksum
PASS: ToolchainInstaller is pure (no shell/UI imports)
```

## What is intentionally NOT done here

- The installer is wired into `FirmwareOperationService.onCreate` (Stage 21)
  and into `LegacyShellBridge.init` (Stage 23, `installToolkit`). Both call
  sites extract the toolkit from `assets/toolchain/manifest.json` into
  `<filesDir>/bin/`.
- The `assetProvider` lambda is the test seam; in production it is backed by
  `AssetManager.open("bin/<name>")` from the `pio` module — wired in
  `LegacyShellBridge.installToolkit` and `FirmwareOperationService.openToolStream`.
- The installer does not handle ABI filtering at extraction time — that is the
  resolver's job. If the manifest declares multiple ABIs for one tool name,
  the caller is expected to provide the right binary via `assetProvider`.
- The manifest ships with real SHA-256 values for every shipped binary
  (generated by `tools/compute-tool-hashes.py`). Two entries (`lpunpack`,
  `simg2img`) keep `sha256=null` because the binaries are not shipped.
